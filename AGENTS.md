# AGENTS.md

## 문서 목적
- 이 문서는 `anglangl` 프로젝트의 장기 운영 기준 문서다.
- 앞으로 Codex와 개발자는 이 문서를 제품의 정체성, 설계 우선순위, 브랜딩 방향, 구현 판단 기준의 1차 기준으로 사용한다.
- 기능 제안, UI 변경, 정보구조 개편, 문구 작성, 데이터 모델 확장, AI 기능 추가 시 모두 이 문서를 먼저 참조한다.
- 일회성 작업 메모가 아니라, 서비스 방향을 일관되게 유지하기 위한 프로젝트 헌장에 가깝다.

## 프로젝트 개요

### 서비스명
- 한글명: `앙글앙글`
- 영문명: `anglangl`

### 한 줄 정의
- AI · LLM 기반 영어 학습 자료 생성 및 공유 플랫폼
- 필요한 영어 학습 자료를 스스로 만들고, 함께 나누는 영어 학습 지원 사이트

### 현재 저장소 성격
- 이 저장소는 Django 기반의 멀티앱 웹 서비스다.
- 현재는 영상 등록, 클립 생성, 학습 이력, 대시보드, 내부 API, 레거시 유틸 앱이 함께 존재한다.
- 앞으로의 방향은 단순 영상 처리 사이트가 아니라, 영어 학습 자료를 생성, 저장, 정리, 재사용, 공유하는 플랫폼으로 수렴해야 한다.
- 기존 영상/클립 기능은 버릴 대상이 아니라, 영어 학습 자료 생성의 핵심 입력 자산으로 재해석해야 한다.

## 서비스 정체성

### `anglangl`이 무엇인가
- `anglangl`은 사용자가 영어 학습 자료를 직접 만들 수 있게 돕는 서비스다.
- 이 서비스의 본질은 "AI가 공부를 대신하는 것"이 아니라 "사용자가 자신의 학습 자산을 더 쉽게 만들고 축적하게 하는 것"이다.
- 따라서 생성된 결과물은 일회성 답변이 아니라 재사용 가능한 학습 자료여야 한다.
- 또한 개인 생성 도구에 머무르지 않고, 다른 사용자의 자료를 탐색하고 가져오고 편집하고 다시 공유할 수 있는 플랫폼이어야 한다.

### 서비스가 지향하는 경험
- 사용자는 필요한 영어 자료를 검색만 하는 사람이 아니라, 직접 만드는 사람이 되어야 한다.
- 사용자는 영상, 대사, 문장, 단어, 주제, 시험 목적, 회화 상황 등 다양한 입력으로 자료를 생성할 수 있어야 한다.
- 생성된 자료는 저장되고, 정리되고, 다시 열람되고, 편집되고, 복제되고, 공유될 수 있어야 한다.
- 플랫폼은 콘텐츠 소비 공간보다 학습 자산이 자라나는 정원, 포도밭, 자료 생태계에 가까워야 한다.

## 브랜드 의미

### 이름의 의미
- `앙글앙글`은 포도가 주렁주렁 많이 열려 있는 이미지에서 출발한 이름이다.
- 이 이름은 영어 학습 자료가 풍성하게 열려 있는 상태를 상징한다.
- 필요한 자료가 계속 나오고, 학습 자원이 풍부하게 맺히고, 사용자가 직접 그 열매를 키워내는 생태계를 의미한다.
- `anglangl`은 일반 영어 단어처럼 해석하는 이름이 아니라, 고유한 브랜드명으로 취급한다.

### 브랜드가 전달해야 하는 감각
- 풍성함
- 생성성
- 연결성
- 축적성
- 학습의 성장감
- 지적이지만 차갑지 않은 친근함

## 핵심 사용자
- 영어를 처음 체계적으로 공부하려는 학습자
- 리스닝, 쉐도잉, 회화, 시험 대비 등 특정 목적의 자료가 필요한 학습자
- 영화, 드라마, 유튜브, 뉴스, 인터뷰 같은 실제 콘텐츠 기반 자료를 만들고 싶은 사용자
- 자신이 만든 영어 학습 자료를 장기적으로 정리하고 공유하고 싶은 사용자
- AI를 활용해 개인 맞춤형 영어 학습 워크플로우를 만들고 싶은 사용자

## 핵심 문제와 해결 방식

### 해결하려는 문제
- 시중 영어 자료는 획일적이거나 내 수준과 목적에 맞지 않는 경우가 많다.
- 필요한 자료가 없으면 직접 만드는 데 시간이 너무 오래 걸린다.
- 좋은 자료를 만든 사람과 그 자료가 필요한 사람이 잘 연결되지 않는다.
- 학습 자료가 일회성으로 소비되고 자산으로 축적되지 않는다.

### `anglangl`의 해결 방식
- AI/LLM을 이용해 자료 생성 비용을 크게 낮춘다.
- 사용자가 입력한 콘텐츠와 목적에 맞춰 맞춤형 자료를 생성한다.
- 생성 결과를 편집, 저장, 라이브러리화, 공개, 복제할 수 있게 한다.
- 다른 사용자의 자료를 검색, 탐색, 가져오기, 재활용할 수 있게 한다.
- 개인 학습 도구와 커뮤니티형 자료 플랫폼을 하나의 흐름으로 연결한다.

## 기능 우선순위

### 최우선
- 사용자가 영어 학습 자료를 쉽게 생성할 수 있어야 한다.
- 생성 결과를 저장하고 다시 열 수 있어야 한다.
- 자료가 사용자 라이브러리 안에서 구조적으로 정리되어야 한다.

### 높은 우선순위
- 자료 공개/비공개 및 공유
- 다른 사용자의 자료 탐색, 검색, 복제
- 영상, 문장, 스크립트 기반 생성 워크플로우 강화

### 중기 우선순위
- 목적별 템플릿 체계 고도화
- 컬렉션, 큐레이션, 커뮤니티 기능
- 추천 시스템, 개인화, 학습 흐름 자동화

### 낮은 우선순위 또는 지양
- 영어 학습과 무관한 범용 콘텐츠 플랫폼화
- 단순 소셜 피드 중심 확장
- 학습 자산 축적과 무관한 장식성 기능

## 제품 철학
- 자료는 주어지는 것이 아니라 사용자가 만들 수 있어야 한다.
- 좋은 학습 자료는 많이 존재해야 하고, 쉽게 발견되어야 하며, 다시 활용될 수 있어야 한다.
- AI는 사용자를 대체하는 도구가 아니라 제작 능력과 학습 효율을 확장하는 도구여야 한다.
- 결과물은 채팅 응답이 아니라 학습 자산이어야 한다.
- 플랫폼은 저장소가 아니라 학습 자료가 쌓이고 연결되고 자라나는 생태계여야 한다.
- 딱딱한 교육 서비스가 아니라 풍성하고 계속 열리는 학습 환경을 지향한다.

## UI/UX 가이드

### 기본 방향
- 모던
- 심플
- 또렷함
- 집중감
- 직관적 흐름

### 화면 설계 원칙
- 사용자가 AI 기능을 어렵게 느끼지 않도록 해야 한다.
- 핵심 흐름은 항상 `입력 -> 생성 -> 편집 -> 저장 -> 공유 -> 재사용` 순서가 자연스럽게 드러나야 한다.
- 복잡한 기능이 많아져도 화면 구조는 단순하게 유지해야 한다.
- 생성형 기능은 마법처럼 숨기지 말고, 사용자가 입력값과 결과를 통제할 수 있게 보여줘야 한다.
- 자료는 채팅 버블보다 문서형, 카드형, 라이브러리형 구조가 더 중요하다.
- 영상/클립 기능은 독립 기능이 아니라 학습 자료 생성의 입력 소스로 보여야 한다.

### 정보구조 원칙
- 자료 유형, 학습 목적, 소스 콘텐츠, 저장 상태, 공개 상태가 명확히 구분되어야 한다.
- 사용자는 자신이 만든 자료와 가져온 자료를 쉽게 구분할 수 있어야 한다.
- "생성"과 "보관"과 "탐색"이 섞이지 않게 네비게이션을 설계한다.
- 자료 상세 화면에서는 재생성, 편집, 저장, 복제, 공유 액션이 분명해야 한다.

### 인터랙션 원칙
- AI 생성 중 상태는 명확해야 한다.
- 실패 시 이유와 다음 행동을 알려줘야 한다.
- 비동기 작업은 조용히 사라지지 말고 상태 추적이 가능해야 한다.
- 긴 생성 작업이나 미디어 처리 작업은 진행 상태, 결과, 오류 메시지를 구조적으로 노출해야 한다.

## 콘텐츠 및 카피라이팅 방향

### 톤앤매너
- 현대적이고 똑똑하지만 친근한 톤을 유지한다.
- 지나치게 장난스럽거나 과도하게 가벼운 교육 서비스 톤은 피한다.
- 반대로 전통 학원처럼 딱딱하고 권위적인 말투도 피한다.
- 문구는 짧고 분명해야 하며, 브랜드 의미가 자연스럽게 드러나야 한다.

### 카피라이팅 원칙
- 한국어를 기본으로 한다.
- 필요 시 짧은 영문 태그라인을 병기할 수 있다.
- "AI가 대신 해준다"보다 "내가 더 쉽게 만들 수 있다"는 메시지를 우선한다.
- "자료 생성", "정리", "저장", "공유", "다시 활용" 같은 동사를 자주 사용한다.
- 학습자의 자율성과 생산성을 강조한다.

### 피해야 할 표현
- 단순 숙제 대행처럼 보이는 문구
- 무조건 쉬움만 강조하는 가벼운 광고 문구
- 학습 없이 정답만 주는 서비스처럼 보이는 표현

## 콘텐츠 방향
- 실제 영어 콘텐츠를 학습 자산으로 바꾸는 경험을 강화한다.
- 영화, 드라마, 유튜브, 뉴스, 인터뷰, 강의, 일상 회화 자료를 적극적으로 다룬다.
- 단어, 문장, 문맥, 발화 상황, 뉘앙스, 반복 학습 포인트가 자료 안에 살아 있어야 한다.
- 사용자가 만든 자료는 문서형 자산으로 축적될 수 있어야 한다.
- 다른 사용자의 자료를 그대로 소비하는 것보다, 가져와 내 용도에 맞게 재구성하는 흐름이 중요하다.

## 현재 코드베이스를 해석하는 기준

### 현재 구조
- 현재 저장소에는 `videos`, `clips`, `study`, `workers`, `dashboard`, `internal_api`, `dramaNlearn` 등이 있다.
- 이 구조는 과거 영상 처리 중심 기능이 누적된 결과이지만, 앞으로는 모두 영어 학습 자료 생성 플랫폼의 자산으로 재배치해야 한다.

### 해석 원칙
- `videos`와 `clips`는 단순 미디어 관리가 아니라 학습 자료 생성의 원천 콘텐츠 저장소다.
- `study`는 학습 이력 자산이다.
- `internal_api`는 내부 도구, MCP, 자동화 파이프라인과 연결되는 확장 포인트다.
- 새 기능은 기존 앱 위에 얹더라도, 최종 사용자 경험은 하나의 `anglangl` 서비스처럼 느껴져야 한다.

## 개발 의사결정 원칙
- 기능을 추가할 때마다 "이 기능이 사용자가 영어 학습 자료를 더 쉽게 만들고, 더 잘 축적하고, 더 잘 공유하게 하는가?"를 먼저 묻는다.
- 영어 학습 자료 생성, 관리, 공유라는 중심축에서 벗어나는 범용 플랫폼화는 피한다.
- 새 모델, 새 화면, 새 API는 자료의 생성, 구조화, 검색, 재사용과 어떤 관계가 있는지 분명해야 한다.
- 기능을 늘리기보다 흐름을 선명하게 만드는 쪽을 우선한다.
- 저장되는 데이터는 가능하면 재생성 가능한 응답이 아니라 재사용 가능한 자산 구조를 가져야 한다.
- AI 기능은 데모성 답변보다 편집 가능한 출력물, 템플릿, 히스토리, 복제 가능성에 초점을 둔다.
- 공개 기능을 만들 때는 공유 편의성과 저작권, 개인정보, 학습 맥락을 함께 고려한다.

## 데이터 및 도메인 설계 원칙
- "메시지"보다 "학습 자료"를 우선 개념으로 둔다.
- 생성 결과는 유형이 드러나는 엔티티로 저장하는 것을 우선한다.
- 단일 거대 텍스트 필드보다 목적이 드러나는 구조화 필드를 선호한다.
- 개인 자산과 공유 자산의 경계를 명확하게 둔다.
- 원본 소스와 가공 결과, 사용자 메모, 생성 이력, 공유 상태를 분리해 관리하는 방향을 선호한다.

## AI/LLM 기능 설계 원칙
- AI는 보조 저자(co-author)처럼 동작해야 한다.
- 입력값, 프롬프트 의도, 생성 결과, 후편집 흐름이 추적 가능해야 한다.
- 결과물은 복사해서 끝나는 답변이 아니라 저장, 수정, 재구성, 공유가 가능한 형태여야 한다.
- 같은 자료를 다른 난이도, 다른 목적, 다른 형식으로 다시 생성할 수 있어야 한다.
- 프롬프트 내부 구현보다 사용자 목적과 출력 구조를 먼저 설계한다.
- 모델 종속성은 서비스 전면에 드러내기보다, 사용자 경험 안에서는 "자료 생성 도구"로 추상화한다.

## 커뮤니티 및 공유 설계 원칙
- 공유는 보여주기용 피드보다 재사용 가능한 자료 단위 중심으로 설계한다.
- 다른 사용자의 자료를 복제해 내 라이브러리로 가져오는 흐름이 중요하다.
- 단순 좋아요보다 컬렉션, 복제, 활용도, 학습 목적 태깅 같은 신호를 더 중요하게 다룬다.
- 공개 자료의 품질 맥락이 드러나도록 소스, 목적, 난이도, 설명을 함께 관리한다.

## 기술 운영 원칙

### 백엔드
- Django 멀티앱 구조는 유지하되, 사용자 경험은 하나의 제품처럼 통합해야 한다.
- 비동기 작업은 Celery와 Redis 흐름을 존중한다.
- 장시간 작업은 작업 상태 추적과 오류 메시지 구조를 우선한다.
- 내부 자동화 연동은 `internal_api` 또는 명확한 서비스 레이어를 통해 구성한다.

### 프론트엔드
- 템플릿 기반이더라도 제품 일관성이 유지되어야 한다.
- 네비게이션은 기능 나열보다 사용자 과업 기준으로 정리한다.
- 생성 화면, 상세 화면, 라이브러리 화면, 탐색 화면의 역할이 겹치지 않게 한다.

### 레거시 처리
- 서비스 정체성과 맞지 않는 과거 기능은 단계적으로 흡수하거나 제거한다.
- 레거시 제거 시에는 코드, 데이터, 파일, 메뉴, 문서를 함께 정리해 잔존 흔적을 최소화한다.

## 네이밍 원칙
- 제품 전면 브랜드는 `앙글앙글` / `anglangl`을 사용한다.
- `anglangl`은 고유명사로 취급한다.
- 기능명은 가능한 한 학습 목적이 드러나게 짓는다.
- 내부 코드 네이밍도 모호한 범용어보다 역할 중심으로 짓는다.
- `save`, `data`, `tool` 같은 뭉뚱그린 이름보다 `study_material`, `shadowing_note`, `listening_set`, `source_clip` 같은 의미 있는 이름을 선호한다.

## Codex Work History Policy

### 목적
- 이 프로젝트의 모든 Codex 작업은 파일 기반 히스토리로 남겨야 한다.
- VS Code 또는 Codex 세션이 예기치 않게 종료되더라도, 사용자의 요청과 Codex의 수행 결과를 복구하고 추적할 수 있어야 한다.
- 이 히스토리는 사람도 읽을 수 있어야 하고, 이후 Codex 세션도 참조할 수 있어야 한다.

### 저장 위치
- 프로젝트 루트 아래 `codex-history/` 디렉터리를 사용한다.
- 디렉터리가 없으면 Codex가 생성한다.
- 기존 히스토리 파일은 절대 자동 삭제하지 않는다.
- 삭제, 정리, 압축, 이동은 사용자만 할 수 있다.

### 생성 규칙
- 사용자의 새 작업 요청이 들어오면, Codex는 실제 작업 시작 전 또는 늦어도 작업 직후 반드시 히스토리 파일을 생성해야 한다.
- 사용자가 "새로운 작업"을 요청한 경우, 기존 히스토리 파일을 덮어써 이어 쓰지 말고 해당 작업을 위한 새 히스토리 파일을 추가 생성하는 것을 기본으로 한다.
- 히스토리 파일은 기존 파일을 덮어쓰지 않고 새 파일을 계속 추가하는 방식으로 누적 저장한다.
- 파일 형식은 Markdown `(.md)`를 사용한다.
- 이후 작업 시 필요하면 `AGENTS.md`와 `codex-history/`를 함께 참고해 맥락을 유지한다.

### 파일명 규칙
- 파일명 형식은 반드시 `{project_name}-{YYYYMMDD}-{seq}.md` 를 따른다.
- 이 프로젝트의 `project_name`은 항상 `anglangl`이다.
- 예시:
- `anglangl-20260315-001.md`
- `anglangl-20260315-002.md`
- `anglangl-20260316-001.md`

### 순번 규칙
- 날짜별로 `001`부터 시작한다.
- 같은 날짜의 새 작업은 `002`, `003`처럼 증가한다.
- 새 파일을 만들기 전에 해당 날짜의 기존 파일 목록을 확인해 다음 순번을 계산한다.
- 순번은 항상 3자리 zero padding을 사용한다.

### 히스토리 파일 본문 규칙
- 각 히스토리 파일은 아래 구조를 따른다.
- `# Codex Work History`
- `## 1. Basic Info`
- `## 2. User Request`
- `## 3. Codex Understanding`
- `## 4. Work Performed`
- `## 5. Files Changed`
- `## 6. Output / Result`
- `## 7. Decisions`
- `## 8. Pending / Next Actions`
- `## 9. Notes for Future Codex Sessions`
- `## 1. Basic Info`의 `Status` 값은 `requested / in_progress / completed / partial / blocked` 중 하나를 사용한다.

### 기록 원칙
- 사용자의 요청 내용, Codex의 해석, 실제 수행 작업, 변경 파일, 결정 사항, 보류 사항, 다음 작업 맥락을 남긴다.
- 완료되지 않은 작업은 `partial` 또는 `blocked`로 명확히 기록한다.
- 추측보다 실제 수행 내용 중심으로 적는다.
- 사람이 빠르게 읽고 복구할 수 있도록 간결하고 명확하게 적는다.
- "무엇을 요청받았는지 / 무엇을 했는지 / 무엇이 바뀌었는지 / 다음에 무엇을 해야 하는지"가 분명해야 한다.
- 작업 실패, 중단, 보류도 숨기지 말고 기록한다.
- 히스토리 누락이 없도록 이것을 기본 작업 절차로 삼는다.
- 사용자가 별도로 삭제를 요청하지 않는 한 어떤 히스토리 파일도 삭제하지 않는다.

## Codex가 이 프로젝트에서 반드시 지켜야 할 기준
- 이 프로젝트를 단순 영상 사이트, 단순 AI 챗봇, 단순 문서 저장소로 해석하지 않는다.
- 모든 제안과 구현은 `영어 학습 자료 생성 및 공유 플랫폼`이라는 정체성에 맞춰 판단한다.
- 새 기능을 만들 때는 생성, 편집, 저장, 공유, 재사용 중 어디에 기여하는지 분명히 한다.
- UI 문구와 정보구조는 브랜드 톤과 학습 목적에 맞게 유지한다.
- 기능이 멋져 보여도 학습 자료 축적 생태계에 기여하지 않으면 우선순위를 낮춘다.
- 답변형 인터페이스를 추가하더라도 최종 산출물이 자산화되는 구조를 우선한다.
- 기존 구현을 수정할 때는 레거시 호환성과 현재 사용자 흐름을 함께 고려한다.
- 디자인을 제안할 때는 검정 중심, 보라/녹색 포인트, 선명하고 현대적인 인상을 유지한다.
- `앙글앙글`의 핵심 이미지는 "풍성하게 열리는 학습 자료 생태계"라는 점을 잊지 않는다.
- 모든 새 작업에서는 `AGENTS.md`와 `codex-history/`를 함께 참조하고, 히스토리 파일 기록을 기본 절차로 수행한다.

## 문서 갱신 원칙
- 이 문서는 기능 목록보다 정체성과 판단 기준이 흔들릴 때 먼저 갱신한다.
- 새로운 대형 기능 축이 생기면, 구현 전에 이 문서에 그 기능이 제품 정체성과 어떻게 연결되는지 반영한다.
- 세부 기술 계획은 별도 문서로 분리할 수 있지만, 제품 철학과 핵심 원칙은 이 문서에 남긴다.
